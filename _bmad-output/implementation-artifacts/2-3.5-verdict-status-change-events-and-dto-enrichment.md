# Story 2.3.5: Verdict Status-Change Events and DTO Enrichment

Status: done

## Story

As a User,
I want to be notified when my partner's risk status changes between searches,
so that I can act on deteriorating or improving partner health without manually re-checking.

## Acceptance Criteria

1. **Given** a search completes and a verdict is persisted, **When** a previous verdict exists for the same tax_number and tenant_id, **And** the new verdict status differs from the previous one, **Then** a `PartnerStatusChanged` event is published containing verdictId, tenantId, previousStatus, and newStatus.

2. **Given** a search completes and a verdict is persisted, **When** no previous verdict exists for this tax_number+tenant_id (first-ever search), **Then** NO PartnerStatusChanged event is published. The first search establishes the baseline, not a "change."

3. **Given** a search completes and a verdict is persisted, **When** a previous verdict exists but the status is UNCHANGED (e.g., both RELIABLE), **Then** NO PartnerStatusChanged event is published. Only real status transitions trigger notifications.

4. **Given** the VerdictEngine (from Story 2-3) produces a VerdictResult containing a riskSignals list, **When** the SearchResult is constructed, **Then** the riskSignals are included in the SearchResult and propagated to the VerdictResponse DTO so the frontend can display the specific reasons for a verdict.

5. **Given** the VerdictResponse DTO, **When** it is serialized to JSON, **Then** it includes a `riskSignals` array field (e.g., ["PUBLIC_DEBT_DETECTED", "SOURCE_UNAVAILABLE:nav-debt"]) and the status/confidence fields use the enum name strings matching the OpenAPI contract.

6. **Given** the PartnerStatusChanged event is published, **Then** it is published OUTSIDE the TX2 transaction boundary (after commit) to ensure the verdict is actually persisted before downstream listeners (notification module) attempt to read it.

## Dev Notes

### Critical Context -- What This Story Changes

This story adds the **notification trigger pipeline** on top of Story 2-3's VerdictEngine. When a partner's risk status changes (e.g., from RELIABLE to AT_RISK), the system publishes a domain event that downstream modules (notification, watchlist monitoring) will consume.

It also enriches the API response with the specific reasons for a verdict (riskSignals), enabling the frontend (Story 2-4) to display "Why is this partner At-Risk?" details.

**The call flow after this story:**

```
ScreeningService.search(taxNumber)
    -> TX2:
        -> updateSnapshotData(...)
        -> parse + evaluate (Story 2-3)
        -> createVerdict(id, status, confidence)
        -> writeAuditLog
    -> findPreviousVerdict(taxNumber, tenantId)         <-- NEW
    -> if status changed: publish PartnerStatusChanged   <-- NEW
    -> publish PartnerSearchCompleted
    -> return SearchResult with riskSignals              <-- ENRICHED
```

### DANGER ZONES -- Common LLM Mistakes to Avoid

1. **DO NOT publish PartnerStatusChanged inside TX2.** If the transaction rolls back after the event is published, downstream listeners will try to read a verdict that does not exist. Publish AFTER the TransactionTemplate callback completes successfully.

2. **DO NOT publish an event on the first-ever search.** No previous verdict = no "change." The first search establishes the baseline.

3. **DO NOT compare confidence for status-change detection.** Only compare VerdictStatus (RELIABLE/AT_RISK/INCOMPLETE/TAX_SUSPENDED/UNAVAILABLE). Confidence changes (FRESH to STALE) are NOT status changes.

4. **DO NOT change VerdictEngine or its tests.** Story 2-3 delivered a locked-down pure function. This story only touches ScreeningService, ScreeningRepository, and DTOs.

5. **DO NOT break the existing PartnerSearchCompleted event.** It still fires after every search. PartnerStatusChanged is an ADDITIONAL event that fires only on status transitions.
