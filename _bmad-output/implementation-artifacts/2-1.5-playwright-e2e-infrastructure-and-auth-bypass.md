# Story 2.1.5: Playwright E2E Infrastructure & Auth Bypass

Status: done

## Story

- As a developer, I want Playwright E2E tests running in CI so that cross-boundary integration bugs (CORS, CSRF, JWT algorithm mismatch, cookie handling, API path construction) are caught before merge — not discovered manually in the browser.

## Background

Playwright E2E was deferred in Stories 1.4 and 1.5. The Epic 1 retrospective mandated: "Every Epic 2 story must ship with unit, integration, AND E2E tests. No deferrals." Story 2.1 implementation revealed 8 cross-boundary bugs that unit/integration tests fundamentally cannot catch (see Dev Notes below). This story bootstraps the infrastructure so all subsequent stories include E2E coverage.

## Acceptance Criteria

1. **Playwright installed and configured** — `playwright.config.ts` with base URL pointing to local backend+frontend, browser(s) configured (Chromium minimum), test directory structure established.
2. **`@Profile("test")` auth bypass endpoint** — `POST /api/test/auth/login` that accepts `{ email, role }` and returns an `auth_token` HttpOnly cookie with a valid JWT for a seeded test user. Only active when `SPRING_PROFILES_ACTIVE=test`. Returns 404 in production.
3. **Test data seeding** — Flyway test migration or `@BeforeAll` setup that creates a test tenant + test user for E2E tests. Idempotent (safe to run repeatedly).
4. **Login → Dashboard → Search E2E test** — Playwright test that: authenticates via test bypass → navigates to `/dashboard` → enters a tax number → clicks "Start screening" → verifies skeleton loading appears → verifies verdict response renders.
5. **CI integration** — GitHub Actions workflow runs Playwright tests after backend+frontend are started. Tests run against real backend (Testcontainers PostgreSQL) + real frontend (Nuxt dev or preview mode).
6. **All existing tests still pass** — No regressions in unit or integration tests.

## Dev Notes

### Bugs from Story 2.1 that E2E tests would have caught

| Bug | Root Cause | Category |
|---|---|---|
| CORS 403 | No `CorsConfigurationSource` configured | Cross-origin |
| CSRF 403 on POST | CSRF enabled with no token from SPA | Cross-origin |
| Doubled `/api/v1/api/v1/` path | `apiBase` included `/api/v1`, endpoints also included it | Config mismatch |
| `credentials: 'include'` missing | `$fetch` doesn't send cookies cross-origin by default | Cross-origin |
| HS512 vs HS256 JWT mismatch | `TokenProvider` signs HS512, `NimbusJwtDecoder` defaults to HS256 | Config mismatch |
| `user_id` claim missing from JWT | `TokenProvider.createToken()` didn't include it, controller expected it | Contract mismatch |
| `isAuthenticated` always false | Checked `!!state.token` but HttpOnly cookies aren't readable by JS | Design mismatch |
| `/me` response array vs object | Controller returns object, but defensive handling needed | Contract |

### Technical Approach

- **Auth bypass pattern**: `@Profile("test")` controller that accepts email+role, looks up or creates user, issues JWT cookie. Same cookie mechanism as production OAuth flow, just without the Google/Microsoft redirect dance.
- **No SSO provider dependency in tests**: Avoids Google/Microsoft bot detection, rate limiting, and credential management in CI.
- **Playwright config**: Use `webServer` option to auto-start backend+frontend before tests run.
