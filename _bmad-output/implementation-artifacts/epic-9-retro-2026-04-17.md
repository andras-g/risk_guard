## What Went Well

### Technical Breakthroughs

1. **Atomic dedup with `ON CONFLICT DO NOTHING`** (Story 9.2)
   - Replaced check-then-insert with single atomic INSERT + return-value inspection
   - Solved TOCTOU race and full-batch-rollback issue in one pattern
   - Should be reused in every dedup path going forward

2. **Popover hoisting with `appendTo="body"` + z-index via `:pt.root.class`** (Story 9.5, AC#5)
   - Three-fold regression (per-row popover leak + DataTable `overflow-hidden` clipping + sidebar drawer stacking) fixed atomically
   - Single Popover instance at page root, targeted by `_tempId`, z-index ≥ 60
   - Pattern documented in Story 9.5 review; worth extracting as a repo convention

3. **Multi-layer AI JSON schema** (Story 9.6)
   - Replaced single-KF-code response with structured `[{layer, kfCode, weightEstimateKg, unitsPerProduct, description}, ...]`
   - Enables multi-component pre-population in both Registry editor and bootstrap dialog
   - Lays the foundation for Epic 10's seed-based tenant onboarding

4. **Lazy credential initialization** (Story 9.3)
   - `GoogleCredentials.getApplicationDefault()` deferred to first classify call via `credentialsUnavailable` flag
   - Preserves the non-blocking Spring startup promise when ADC missing (local dev)
   - Pattern reusable for any future GCP integration

### Process Wins

- **Zero production incidents** across 6 stories in 3 days, including AI integration + compliance-critical math + new export format (OKIRkapu XML)
- **R2 code reviews caught critical compliance bugs** that R1 missed — the float-precision issue in `KgKgyfNeMarshaller.floatValue()` (7 significant digits on weight aggregates) would have produced incorrect EPR filings at production scale
- **Sprint-status discipline** — Epic 6 retro action item P2 ("sprint-status.yaml update as DoD") was followed consistently; only the status-flip timing on 9.3/9.4/9.5 lagged (closed on 2026-04-17)

## Significant Discovery — Workflow Paradigm Shift for Epic 10

**Found during this retrospective, not during Epic 9 execution:** the workflow that Epic 9 built is being restructured for Epic 10.

**Today's product-direction conversation with András concluded:**
- **`Anyagkönyvtár` as a standalone menu item will be removed** in Epic 10. The `epr_material_templates` table stays but becomes a "building block" table referenced by Registry components via a nullable FK (`products_components.material_template_id`).
- **Menu reorders:** `Termék–Csomagolás Nyilvántartás` → `Negyedéves EPR bejelentés`. EPR page becomes the final computation step, not the data-entry starting point.
- **EPR filing becomes product-first, not template-first:** aggregation walks invoice lines → resolve to registered products (via `VTSZ + description` key) → walk components × `items_per_parent` ratios → sum by KF code. No more manual template-quantity entry.
- **Initial Registry population is AI-powered** — on tenant onboarding, the classifier runs batch mode over all unique `(VTSZ + description)` pairs and suggests full packaging stacks.
- **Compliance approach chosen: option (C)** — live Registry state drives calculation; submitted XMLs preserved read-only in `epr_exports` audit log (no snapshot in Registry itself).

**Impact on Epic 10:** Story 10.1 must bundle schema bump + menu restructure + `Anyagkönyvtár` absorption + data migration. **L size, not M.** Plan accordingly.

**Epic 10 Update Required:** YES — the 9 stories sketched in today's conversation need formal epic document + AC breakdown before Story 10.1 starts. Bob to run `bmad-create-epics-and-stories` or equivalent sprint-planning session.
