## What Went Well

### 1. Manual Testing Found Only 1–2 Bugs
Epic 4's post-epic manual testing session caught 13 integration-level bugs. Epic 5 produced 1–2. This is partly attributable to the maturing review process absorbing integration issues earlier (e.g., Story 5.2's real-time per-row computation and summary visibility were caught in R2 review, not manual testing). Partly it reflects the nature of the stories — output-format work (CSV bytes, toast text) has smaller UX integration surface than wizard flows or dashboards.

**Caveat:** Epic 6 returns to complex admin UI (live health dashboards, circuit breaker toggles). The manual testing burden is likely to increase again.

### 2. Cleanest Code Reviews in Project History
- Story 5.3 R2: **0 patches** — the first completely clean second-round review in the project.
- Story 5.4: **1 patch total** across all rounds.
- The adversarial review process is genuinely maturing. The team is anticipating the kinds of issues that previously slipped through.

### 3. Mid-Epic Story Absorbed Without Disruption
Story 5.1a (My Audit History) was created and completed on 2026-03-30 — an unplanned addition. The codebase and process were stable enough to absorb it same-day. This reflects improved scaffolding quality from previous foundation stories.

### 4. Deferred Work Discipline
Instead of blocking stories on edge cases or holding findings open, the team consistently moved appropriate items to `deferred-work.md`. The file is now a well-organised log that serves as the input for the Epic 6.0 cleanup pass.

### 5. MOHU Compliance Delivered as a Real Regulatory Artifact
The EPR CSV export is schema-correct for MOHU submission: UTF-8 BOM, Hungarian locale enforcement, correct column headers from the regulations. This is a genuine deliverable, not a demo placeholder.

## What Didn't Go Well

### 1. T1 and T2 Rolled Forward Again
`requireUuidClaim` is now copy-pasted in `EprController`, `ScreeningController`, and `WatchlistController` — four epics after it was first flagged. Epic 6 adds an admin controller, which will be the 4th copy unless T1 is resolved in 6.0. `ScopedValue` migration has a stated hard deadline of "before Epic 6" and is also unresolved.

**Root cause:** Action items without story-file tasks can't be held accountable. See process decision below.

### 2. Backend Test OOM in Story 5.1
The test suite hit an OOM during Story 5.1 backend tests, requiring `-Xmx1g` in `build.gradle`. The flag is now hardcoded without a CI property override. As the test suite continues to grow, this will need revisiting. Added to deferred-work; flagged as an Epic 6 risk.

### 3. Story 5.2 Required Significant R2 Fixes
Story 5.2's second review round caught real-time per-row computation (AC 2), always-visible summary (AC 4), and ESLint a11y errors that would have broken the CI pipeline. These are correctness and accessibility issues that should have been caught in R1 or pre-review. The story was the most complex in the epic (FeeCalculator, filing endpoint, MOHU Gate, Pinia store, real-time UI), which partially explains the extra round.

## Key Decisions

### Process: Retro Action Items Must Live in Story Tasks
A retrospective action item without a named task in a story file has no accountability mechanism. Going forward: any technical action item from a retro that requires story-slot work must be explicitly listed as a task in the designated story. T1 and T2 are the proof case — both survived two retros as notes and were skipped both times.

### Epic 6 Foundation: Everything in Story 6.0
Story 6.0 will carry three pre-feature tasks before any admin UI work starts:
1. **T1** — Extract `requireUuidClaim` to a shared `core` utility
2. **T2** — ScopedValue migration (ThreadLocal → ScopedValue for TenantContext)
3. **Curated maintainability pass** — audit `deferred-work.md`, address all high-confidence / low-risk items including `$nextTick` → `flushPromises()` migration in `filing.spec.ts`

### Manual Testing Philosophy (Reconfirmed)
The post-epic manual session remains the integration quality gate. Epic 5's quiet session is an outcome of story type, not a process change. Manual testing effort will likely increase in Epic 6 given the admin UI complexity.
