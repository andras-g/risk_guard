## What Went Well

### 1. Course Correction Executed Swiftly and Decisively
The 2026-03-12 course correction was the biggest moment of the epic. Both Cegkozlony and NAV multi-query portals were discovered to have bot-protection (Cloudflare Turnstile and reCAPTCHA respectively). Rather than wasting weeks on CAPTCHA bypass approaches, the decision was made immediately: no bot-protection bypass on government portals. Demo Mode + NAV Online Szamla (legitimate API) replaced the scraper strategy.

The pivot was clean: Story 2.2 was rebranded as Demo Mode data layer, Story 2.2.1 was deferred, Story 2.2.2 delivered the real NAV Online Szamla client, and the sprint-status was updated with full context. The sprint-change-proposal document captured the rationale permanently.

### 2. Severity-Gated Reviews Worked Exactly as Designed
In Epic 1, Story 1.3 required 6 review rounds because the reviewer was surfacing LOW findings in every pass. In Epic 2, the severity gate kept things tight. Story 2.4 had 2 review rounds (Round 1: 1 HIGH + 9 MEDIUM/LOW; Round 2: 1 HIGH + 7 MEDIUM/LOW) -- the HIGH findings drove each re-review, not the LOWs. Story 2.5 had a single review round with 6 findings (1 HIGH, 5 MEDIUM) -- all fixed in one pass, status to done.

### 3. Frontend Test Count Grew Systematically (12 -> 107)
Epic 1 ended with 12 frontend tests. Epic 2 ended with 107. Every story added meaningful frontend coverage: VerdictCard (24 tests), ProvenanceSidebar (15 tests), [taxNumber].vue page (12 tests), VerdictCard sentinel tests (4 additional). The co-location convention (*.vue + *.spec.ts in same directory) meant tests were never an afterthought.

### 4. The E2E Story Paid For Itself Immediately
Story 2.1 shipped with 8 cross-boundary bugs (CORS, CSRF, doubled API paths, JWT algorithm mismatch, cookie handling, claim extraction) that unit/integration tests fundamentally could not catch. Story 2.1.5 bootstrapped Playwright and caught them all in one pass. Every subsequent story had E2E coverage from day one.

### 5. The VerdictEngine as a Pure Function
The decision to make VerdictEngine a pure function (no Spring context, no database, static method) was architecturally sound. It enabled 47+ Golden Case regression tests with zero setup overhead. The evaluation priority order (TAX_SUSPENDED > confidence > source availability > risk signals > RELIABLE) was crystal clear and fully testable.

### 6. Legal Audit Trail is Production-Grade
SHA-256 with null-byte separators, DB CHECK constraint on sha256_hash column, HASH_UNAVAILABLE sentinel instead of silent failures, verdict_id FK for full traceability. Story 2.5 went beyond the AC and delivered genuinely court-ready infrastructure.

## What Did Not Go Well

### 1. Story 2.2.1 Assumption Failure: Scraper-First Architecture
The original Epic 2 architecture assumed JSoup-based scraping of Cegkozlony and NAV. Story 2.2 built the scraping infrastructure (CompanyDataAggregator, StructuredTaskScope, Resilience4j circuit breakers) around adapters that were fictional from day one. The underlying assumption -- that these portals were scrapable -- was never validated before coding began.

This is the most significant failure mode: technical architecture built on unverified external system assumptions.

### 2. The scraping Module Was Built and Then Immediately Replaced
Story 2.2 built NavDebtAdapter, ECegjegyzekAdapter, and CegkozlonyAdapter as JSoup scrapers. Story 2.2.2 deleted the entire scraping package and replaced it with the datasource module. The module rename + adapter deletion added significant overhead (1.12: delete 14 source files; 1.10: config key rename; ArchUnit test updates).

### 3. AC6 vs AC9 Contradiction in Story 2.3
Story 2.3 had a genuine specification contradiction: AC6 said "UNAVAILABLE confidence forces INCOMPLETE regardless of data content" while AC9 placed TAX_SUSPENDED check FIRST (before confidence). The implementation correctly followed AC9 intent, but the contradiction was only caught in the AI code review, not in story preparation. This is a story quality issue.

### 4. staleThresholdHours Dead Code
FreshnessConfig had a staleThresholdHours field defined in YAML, risk-guard-tokens.json, and the Java record -- but VerdictEngine never read it. Dead config that existed in 4 places simultaneously. Caught in review but should not have been present at all.

### 5. Cookie Configuration Audit (E4) Carried Over Without Resolution
The Epic 1 retro identified cookie configuration inconsistencies (Secure, SameSite, HttpOnly, maxAge set differently across multiple files). This was marked as action item E4. Epic 2 ended without addressing it. The concern is now two epics old.

### 6. ProvenanceSidebar Test Fixture Time-Sensitivity Bug
A test fixture used a fixed checkedAt timestamp (> 6h old at test runtime) that caused the freshness-guard test to fail based on when the test ran. Tests should use relative timestamps (new Date().toISOString()). This is a test-quality issue.

## Key Insights and Patterns

### Insight 1: Validate External Dependencies Before Coding
Before coding any adapter or integration, verify the target system is accessible and its API is what you think it is. A 30-minute investigation (HTTP request, check response headers) would have revealed Cegkozlony Turnstile and NAV reCAPTCHA before the architecture was built around them.

**Action:** Story dev notes for any external integration must include: "External dependency verified: [evidence of API/protocol accessibility]."

### Insight 2: Review Findings Were Concentrated in Frontend-Backend Boundary
Looking across all Epic 2 stories, the highest-density review findings appeared at the frontend-backend boundary: CORS/CSRF (2.1), PrimeVue v4 Accordion value prop (2.4), requireUuidClaim vs extractUuidClaim (2.4), dashboard stale-verdict redirect race (2.4). These are cross-cutting concerns that no single-domain test catches.

E2E tests are the defense here -- and they worked.

### Insight 3: "Dead Code" Pattern Appears Again
Epic 1 had the ContextGuard.vue dead code problem (delivered but never triggered). Epic 2 had staleThresholdHours (configured but never read). The pattern is: code/config delivered in one story that is intended to be used by a future story, but the connection is never made explicit and the dead reference persists.

### Insight 4: DB-Level Constraints as a Final Defense Layer
Story 2.5 added a CHECK constraint on sha256_hash: . This prevents any application bug from writing an arbitrary string into the legal proof column. The pattern -- enforce invariants at the database level, not just in application code -- should be applied consistently.

### Insight 5: Hash Version Tracking Deferred (Technical Debt)
The HashUtil null-byte separator change in Story 2.5 means pre-fix rows in search_audit_log used unseparated hash inputs. A hash_version column was deferred to Story 6.4. This is documented but real -- any legal audit of pre-fix rows would require knowing which formula to re-verify against.

### Gaps and Preparation Needed Before Epic 3

#### Critical (Must address before Epic 3 starts)

**1. Design System Tokens & Application Shell (Story 3.0a)**
Epic 2's frontend used ad-hoc Tailwind utilities. Story 3.0a must establish the canonical color tokens (Deep Navy #0F172A, Forest Emerald #15803D, Crimson Alert #B91C1C, Amber #B45309) and application shell (persistent sidebar, responsive nav) BEFORE any other Epic 3 stories touch the UI. All subsequent stories inherit the shell.

**2. i18n Infrastructure (Story 3.1)**
Epic 2 added i18n keys under the `screening.*` namespace. Epic 3 introduces watchlist, email, and EPR namespaces. The @nuxtjs/i18n module must be formally configured (not just implicit Nuxt auto-imports) with locale files structured as `frontend/app/i18n/hu/*.json` and `frontend/app/i18n/en/*.json` (per project-context.md). Story 3.1 must come very early.

#### Parallel (Can happen during early Epic 3 stories)

**3. Story 3.0b: Public Landing Page**
Epic 2 verified that the backend search endpoint works end-to-end. The landing page (unauthenticated zero-friction search) can be built in parallel with early watchlist stories since it has no dependency on Epic 3's watchlist infrastructure.

**4. Story 3.2: Email/Password Registration**
The current auth is SSO-only. Email/password registration was deferred from Epic 2 to Epic 3 (correctly). This is early-epic work since it affects auth flows for all subsequent stories.

#### Knowledge Gaps

**5. Resend Email API Integration**
Story 3.8 uses Resend for email alerts. The Resend API key secret exists in Secret Manager (RESEND_API_KEY_STAGING). A developer spike to validate the Resend SDK (email sending with Hungarian locale templates, retry/backoff behavior) before Story 3.8 starts would reduce risk.

**6. Story 3.5 AsyncIngestor in Demo Mode**
The AsyncIngestor (Story 3.5) in demo mode is effectively a no-op (demo fixture data is static). The value is validating the scheduling/retry infrastructure. This story should explicitly test that the scheduler fires, that it respects rate limits, and that failures are logged correctly -- even if the data refresh produces no changes.

### Significant Discovery: Epic 3 Stories 3.0a-3.4 Are Foundational

The epics.md note says: "Stories 3.0a-c are foundational UX stories and 3.1-3.4 are cross-cutting infrastructure stories -- all should be implemented early in Epic 3 as they establish the framework for all subsequent UI and business logic stories."

Epic 2 experience confirms this. If Story 3.6 (Watchlist CRUD) is implemented before Story 3.0a (Design System), the UI will need to be refactored when the design system lands. The story order must be respected.
