## What Went Well

### 1. Smaller Epic Scope Felt Better
Epic 4 was 5 focused stories in 3 days vs Epic 3's 16 stories over 7 days. The clear domain boundary (EPR module, start to finish) made it easier to stay oriented. This scope size is preferred going forward.

### 2. Foundation Story Pattern Continues to Pay Off
Story 4.0 resolved every cross-cutting question before any feature work: table schemas, ArchUnit rules, legislation-verified seed data, i18n namespace. Stories 4.1–4.4 had no scaffolding surprises to work around.

### 3. DagEngine Pure Function Architecture Was an Excellent Decision
The JSON-driven DAG traversal as a pure function (no database calls, no side effects) paid off immediately: cache-friendly, trivially testable, and cleanly extensible. Story 4.3's confidence scoring was additive — just another method with no architectural friction. Story 4.4 plugged in on top without any DagEngine changes.

### 4. HIGH Finding Rate Declining Within Epic
The adversarial review process is maturing. HIGH findings per story: 4.0→4, 4.1→6, 4.2→6, 4.3→3, 4.4→2. Story 4.4 required only one review round — the best result in the project's history.

### 5. Manual Testing Workflow is the Right Quality Gate
Andras's practice of manual testing after every ~4–5 stories caught 13 integration-level issues that no automated test would have found: wizard step 3 backend error, DataTable overflow, language-switch mixed breadcrumb, AutoComplete `[object Object]` display, KF codes appearing identical in override dialog. The review process handles correctness; manual testing handles UX and integration.

### 6. Legislation-Verified Seed Data Is a High-Quality Artifact
The `epr-seed-data-2026.json` was sourced from two authoritative legal documents with correct Hungarian terminology (KF kód, KGyfR díj, MOHU — not NAV). This prevents regulatory errors and will serve as the reference for Story 6.3 (Hot-Swappable EPR JSON Manager).

## What Didn't Go Well

### 1. Task Checkbox Unreliability (P2 Violated — Decision: Deprioritize)
Story 4.2 Task 6.4 was marked `[x]` but the DagEngine integration smoke test was not written — caught as a HIGH finding in review. This was explicitly a P2 action item from Epic 3. However, after discussion: Andras considers the adversarial review process to be the real quality gate, not the checkboxes. All issues were caught. **Decision: task checkbox accuracy is no longer a tracked process item.**

### 2. Story 4.2 Required Three Review Rounds
The only story across all four epics to need a third round. The story was the most complex (DagEngine traversal, Pinia wizard state, PrimeVue Stepper, mobile layout — four systems interacting). Round 3 caught the Stepper level-mapping bug that would have corrupted traversal paths for deposit packaging (stream 12). Proportionate to complexity, but worth noting as a cost signal.

### 3. XSS Vulnerability Slipped Through Pre-Review Checklist
Story 4.3: user-supplied `overrideReason` was concatenated into `v-tooltip` without escaping — caught in R1 review. The pre-review security checklist (P4 from Epic 3) didn't prevent this. The review caught it, but earlier detection would be lower cost.

### 4. `requireUuidClaim` DRY Violation Accumulating
Documented in Story 4.1 as tech debt. Now copy-pasted in `EprController`, `ScreeningController`, and `WatchlistController`. Four stories old. Not blocking, but the three copies can drift silently into security inconsistency.

### 5. Missing Files in Dev Agent Record (Cosmetic but Consistent)
Stories 4.2, 4.3, and 4.4 all had review findings about undocumented modified files (`NamingConventionTest.java`, `nuxt.config.ts`). Minor but means the file-change record is incomplete.

## Key Decisions

### Testing Philosophy (Confirmed)
- Automated tests (unit + integration) catch code correctness. Manual testing by Andras after ~4–5 stories catches UX and integration issues. No additional automation layer warranted — the full test suite already takes ~10–15 minutes.
- The post-epic bugfix session is the expected output of the manual testing pass, not a process failure.

### Task Checkbox Accuracy (Deprioritized)
The adversarial review process is the quality gate. Checkbox accuracy is a nice-to-have, not a reliability signal. P2 is retired from active tracking.

### Epic Scope (Confirmed Preference)
5 stories / ~3 days is the preferred epic size going forward. Smaller, focused domain scope over large cross-cutting epics.
