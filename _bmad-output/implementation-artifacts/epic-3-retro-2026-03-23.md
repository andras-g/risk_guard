## What Went Well

### 1. Foundational Story Ordering Was a Game-Changer
Epic 2's retro called for "Story 3.0a before anything else," and the team followed through. The deliberate sequence — design system (3.0a) → landing page (3.0b) → accessibility (3.0c) → i18n (3.1) → auth (3.2) → feature flags (3.3) — meant that by the time business logic stories (3.6+) started, every cross-cutting concern was locked down. Zero UI refactoring, zero token retrofitting, zero i18n bolt-ons.

### 2. Design System Reached "Showable" Quality
The Safe Harbor design system (Story 3.0a) established PrimeVue 4 Aura as the component foundation with CSS custom property tokens. Combined with the landing page (3.0b), the application went from "developer-grade" to "can be shown to stakeholders and potential users" — a significant product milestone. The token-based architecture means a future design refresh is a token swap, not a rewrite.

### 3. Adversarial Code Review Process Matured
Every story received two adversarial review rounds. The review process caught genuinely critical bugs: Story 3.3 Round 2 discovered that all frontend tier gating was 100% dead code (reading from unpopulated `useIdentityStore`). Story 3.12 caught a TOCTOU race condition on guest rate limiting. These would have been production defects.

### 4. Pattern Propagation Across Stories
Fixes in earlier stories became standard patterns in later ones. The `SELECT ... FOR UPDATE` pattern from Story 3.12's guest session race condition fix was immediately adopted in Story 3.13's refresh token rotation. The injected `java.time.Clock` pattern from Epic 2's action items was used in Stories 3.5, 3.7, 3.12, and 3.13.

### 5. Test Suite Growth Was Phenomenal
Frontend tests grew from ~198 to 500+ across the epic. Every story added meaningful, co-located specs. The axe-core accessibility tests from Story 3.0c catch WCAG regressions automatically. Backend unit tests expanded significantly per story, with MockMvc integration tests for controllers.

### 6. Module Boundary Discipline
The `notification` module was built cleanly — watchlist entries, outbox pattern, email delivery — all behind facades. The `PartnerStatusChanged` event from Epic 2 connected seamlessly to Story 3.7's WatchlistMonitor and Story 3.8's outbox processor. Event-driven decoupling paid off as designed.

### 7. Zero Production Incidents
Despite shipping 16 stories with major infrastructure changes (auth, email delivery, refresh tokens, guest sessions), the epic completed with zero production incidents.

## What Didn't Go Well

### 1. HIGH Code Review Finding Density
40+ HIGH-severity findings across 16 stories means the initial implementation consistently shipped with significant defects. Examples: email normalization missing in both `register()` and `login()` (3.2), entire frontend tier gating dead code (3.3), unsafe cast on public endpoint (3.11), module boundary violation in IdentityRepository (3.12). The review process caught them — but the rate is concerning.

### 2. Dev Agent Propagated Stale Information
The Flyway duplicate `R__e2e_test_data.sql` issue was fixed in commit `6861318` during Story 3.4 (2026-03-18). However, subsequent stories (3.6 through 3.13) continued to copy-paste "pre-existing integration test failure — not caused by this story" into their debug logs without re-verifying. This created a false narrative that integration tests were broken for the entire second half of the epic, when in reality CI was green.

### 3. Task Checkboxes Were Unreliable
Several stories marked tasks as `[x]` completed when the work was not actually done. Story 3.3 had Task 5.3 (`TierGateIntegrationTest.java`) marked complete but the file did not exist. Task checkboxes were not reliable indicators of completion.

### 4. Scope Grew Mid-Epic
Two stories were relocated from Epic 2 (3.11 SEO Stubs, 3.12 Guest Rate Limiting) on the first day of the epic. Story 3.13 (Refresh Token Rotation) was added unplanned mid-sprint to address a security gap. While each addition was justified, 16 stories is a large epic and scope discipline should be monitored.

### 5. ScopedValue Migration Still Unaddressed
The `ScopedValue` migration for `TenantContext` (action T3 from Epic 2) has been carried forward for three consecutive epics without being addressed. The ThreadLocal pattern works but becomes harder to refactor the more code depends on it.

## Key Discussion Points

### CI Pipeline Trust
The team discussed CI pipeline reliability. Andras confirmed that **GitHub Actions CI is currently green** and has been for some time. Investigation revealed the Flyway duplicate was fixed in Story 3.4 (commit `6861318`), but the dev agent continued propagating stale "broken CI" disclaimers in all subsequent story debug logs. This highlights a process gap: the dev agent must verify claims rather than copy-pasting from previous stories.

### Story Completion Gate
The team agreed that story completion should require `./gradlew check` (not just unit tests) passing locally before marking done. This catches Spring context wiring issues, interceptor registration, and security configuration that mocks hide. This gate takes effect immediately as a process improvement.

### E2E Test Expectations
Andras expressed the expectation that at minimum, when pushing a story, e2e tests should be green locally. While running full CI on every story is too slow, a local integration smoke test should be the minimum bar. The coding infrastructure to support this (e.g., establishing a test baseline count) will be done as a foundation story in Epic 4.

### ScopedValue Migration Decision
The team agreed to **explicitly defer** the ScopedValue migration for TenantContext. It will NOT be done in Epic 4 (which introduces a new domain module — too risky to mix with infrastructure refactoring). Hard deadline: must be completed before Epic 6. If not addressed by then, it becomes mandatory.

### Epic 4 Preparation
The team identified that Epic 4 (EPR Material Library & Questionnaire) needs a foundation story to scaffold the `epr` Spring Modulith module and prepare EPR fee table data from legislation. The design system being "showable" also opens the door for early user validation with real accountants — a product decision Alice will explore.
