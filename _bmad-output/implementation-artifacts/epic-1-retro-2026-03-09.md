## What Went Well

### 1. Relentless Code Review Culture
Story 1.3 went through 6 rounds of code review. Every round made the SSO implementation genuinely more secure. By round 6, bugs like `@ApplicationModuleListener` running in a separate transaction were caught — the kind of issue that would have been a production incident.

### 2. 100% Story Completion
All 5 stories delivered and marked done. Acceptance criteria were thorough and every one was met.

### 3. Security-First Mindset Emerged
The multi-tenant isolation pattern (`TenantAwareDSLContext` + `TenantJooqListener` with auto-filtering on SELECT, UPDATE, DELETE) is genuinely robust. It took two rounds to harden, but now every future module inherits it for free.

### 4. Test Coverage Foundation
52 backend tests and 12 frontend tests for a foundation epic. The `TenantIsolationIntegrationTest`, `MandateExpiryIntegrationTest`, and ArchUnit architectural tests prevent regression for the entire project lifetime.

### 5. Architecture Patterns Locked Down
Module layering (`api/domain/internal`), DTO record pattern with `from()` factories, event catalog, i18n conventions — all documented in architecture and enforced in code.

---

## What Didn't Go Well

### 1. Security Findings Were Systemic
CRITICAL security issues in 4 out of 5 stories. These weren't obscure — tenant isolation bypasses, weak JWT secrets, fail-open patterns, dead RBAC enforcement. The dev agent consistently delivered code that looked complete but had fundamental security holes.

### 2. "Delivered but Non-Functional" Pattern
Story 1.4's `ContextGuard.vue` was dead code with local refs never triggered. `TenantSwitcher.vue` wasn't integrated into any layout. Story 1.1 had ArchUnit rules commented out. Features were present in the codebase but didn't actually work.

### 3. Story 1.3 Required 6 Review Rounds
40+ findings across 6 rounds on a single story. The SSO implementation touched security, cookies, JWT, OAuth2 state management, module layering, and PII handling — all in one story. Each fix introduced new issues or revealed pre-existing ones.

### 4. Cookie/Auth Attribute Inconsistencies Were Pervasive
Secure flag, SameSite, HttpOnly, maxAge — set differently across `SuccessHandler`, `IdentityController`, `CookieUtils`, and `HttpCookieOAuth2AuthorizationRequestRepository`. Three stories had findings related to this. No single source of truth for cookie configuration.

### 5. No Deployment Validation
An entire identity and authentication system built and never deployed to a real environment. Everything verified locally only.

---

## Root Cause Analysis

### Review Round Inflation
The review agent operates under a "find 3+ issues" mandate per round. This created unnecessary review cycles — even when code quality was genuinely good, the reviewer was forced to surface findings, often LOW-severity stylistic concerns. This inflated Story 1.3 from an estimated 3-4 rounds to 6 rounds.

**Impact:** Developer frustration, artificial velocity reduction, diminishing returns on later rounds.

**Resolution:** Severity-gated reviews adopted for Epic 2 (see Action Items).

## Decisions Made

### Process Changes for Epic 2

| Decision | Detail |
|----------|--------|
| **Severity-gated code reviews** | Only CRITICAL/HIGH findings trigger another mandatory review round. MEDIUM/LOW are logged as advisory notes but don't block story completion. |
| **Test mandate per story** | Every Epic 2 story must ship with unit, integration, AND E2E tests. No deferrals. |
| **Story 2.1 as reference implementation** | Extra care on `screening` module patterns — it's the template all future modules copy. |

### Technical Decisions

| Decision | Detail |
|----------|--------|
| **Bootstrap GCP before Epic 2** | Run `terraform apply`, provision staging, deploy Epic 1 code as baseline before starting any Epic 2 stories. |
| **`@Profile("test")` auth bypass for E2E** | Playwright tests authenticate via a test-only endpoint that issues JWTs for seeded users. No SSO provider dependency in tests. Avoids Google/Microsoft bot detection risk. |
| **Playwright in Story 2.1** | Bootstrap Playwright infrastructure as a subtask of Story 2.1. Twice deferred (Stories 1.4, 1.5) — no more deferring. |
| **Story 2.6: Email/Password Registration** | New story added to Epic 2 for email/password auth (ADR-5 "fallback"). Real product feature, not just testing infrastructure. |

### Technical Findings from Research

- **NAV (National Tax Authority):** JSON exports available via 2025 Open Data Portal — structured API, not HTML scraping. Significantly reduces fragility risk.
- **e-Cegjegyzek (Company Registry):** HTML-based, requires JSoup parsing. DOM inspection recommended before adapter implementation.
- **Playwright:** Reserved for dynamic/JS-heavy sources only. Not needed for MVP's primary adapters (NAV JSON + e-Cegjegyzek JSoup).
