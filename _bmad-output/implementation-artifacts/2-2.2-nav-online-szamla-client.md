## Story

As a Developer,
I want the data source adapter selection driven by the `riskguard.data-source.mode` property with enriched, realistic demo data and a clear NAV Online Számla extension point,
so that the demo mode delivers a compelling, scenario-rich experience and the codebase has a clean architecture for real NAV adapters when credentials become available.

## Acceptance Criteria

1. **Given** the `riskguard.data-source.mode` property set to `demo`, `test`, or `live`, **When** the Spring context starts, **Then** a `@Configuration` class conditionally creates the appropriate `CompanyDataPort` bean(s): `demo` → `DemoCompanyDataAdapter`; `test`/`live` → reserved for future NAV adapters (not yet implemented in this story).

2. **Given** the existing `scraping` module with fictional JSoup-based adapters (`NavDebtAdapter`, `ECegjegyzekAdapter`, `CegkozlonyAdapter`), **When** this story is complete, **Then** the module is renamed to `datasource` (package `hu.riskguard.datasource`) matching the corrected architecture, the fictional adapters are replaced by `DemoCompanyDataAdapter` in `datasource.internal.adapters.demo`, and all JSoup dependencies are removed.

3. **Given** the `DemoCompanyDataAdapter` in the `datasource.internal.adapters.demo` package, **When** it is invoked by the `CompanyDataAggregator`, **Then** it returns realistic Hungarian company data from in-memory Java fixtures — covering at minimum these verdict scenarios: (a) debt-free/reliable, (b) has public debt/arrears, (c) missing tax filings, (d) insolvency proceedings active, (e) suspended tax number — with at least 5 distinct companies.

4. **Given** the demo fixtures, **When** they provide invoice data for a company, **Then** each company includes 20-50 invoices per quarter with realistic line items containing: product descriptions (Hungarian), VTSZ codes, quantities, unit prices, VAT rates — covering both `OUTBOUND` (company's sales) and `INBOUND` (company's purchases) directions — seeding the future EPR module's `QueryInvoiceDigest`/`QueryInvoiceData` needs.

5. **Given** a stub package `datasource.internal.adapters.nav` exists, **When** a developer inspects it, **Then** it contains a `package-info.java` referencing ADR-6 as the future extension point for the NAV Online Számla XML/JAXB integration, plus stub interfaces/records that define the NAV client contract: `NavOnlineSzamlaClient` (service interface with `queryTaxpayer`, `queryInvoiceDigest`, `queryInvoiceData` method signatures), `TaxpayerInfo`, `InvoiceSummary`, `InvoiceDetail` records.

6. **Given** the `test` or `live` mode is configured, **When** the application starts and no NAV adapter bean exists, **Then** the application fails fast at startup with a clear error message (e.g., `"Data source mode '${mode}' requires NAV adapter beans which are not yet implemented. Use 'demo' mode or implement NavOnlineSzamlaAdapter."`) rather than silently falling back to demo data.

7. **Given** the `CompanyDataAggregator` currently injects `List<CompanyDataPort>`, **When** the adapter selection switches to profile-based, **Then** the aggregator still receives only the active adapters for the current mode — in `demo` mode it receives `[DemoCompanyDataAdapter]`, and the parallel `StructuredTaskScope` orchestration, Resilience4j circuit breakers, and tenant propagation patterns from Story 2.2 are preserved.

8. **Given** the existing `ScreeningService` calls `ScrapingService.fetchCompanyData()`, **When** the module is renamed, **Then** `ScreeningService` calls `DataSourceService.fetchCompanyData()` (the renamed facade), and all existing integration tests pass with the new module name.

9. **Given** the `SnapshotDataParser` from Story 2.3, **When** it receives demo fixture data, **Then** the JSONB structure produced by `DemoCompanyDataAdapter` matches the expected adapter key format — using adapter name `"demo"` with fields `available`, `hasPublicDebt`, `taxNumberStatus`, `hasInsolvencyProceedings`, `companyName`, `registrationNumber`, `debtAmount` — so the VerdictEngine produces correct verdicts from demo data without parser changes.

10. **Given** the completed story, **When** a full search flow is executed in demo mode, **Then** the end-to-end flow works: `ScreeningController` → `ScreeningService` → `DataSourceService` → `CompanyDataAggregator` → `DemoCompanyDataAdapter` → `SnapshotDataParser` → `VerdictEngine` → persisted verdict with correct status (RELIABLE/AT_RISK/INCOMPLETE/TAX_SUSPENDED depending on the demo company searched).

## Dev Notes

### Critical Context -- What This Story Changes

This story performs a **structural refactoring** of the data acquisition layer plus **enrichment of demo fixtures** for a compelling demo-first MVP. It does NOT add real NAV API integration yet -- it prepares the clean architecture for it.

Three major changes:

1. **Module rename**: `hu.riskguard.scraping` becomes `hu.riskguard.datasource`. The old JSoup-based fictional adapters (NavDebtAdapter, ECegjegyzekAdapter, CegkozlonyAdapter) are deleted and replaced by a single `DemoCompanyDataAdapter` with rich fixtures.

2. **Profile-based adapter selection**: A `riskguard.data-source.mode` property (demo/test/live) controls which `CompanyDataPort` beans are created. Demo mode uses in-memory fixtures. Test/live modes will use real NAV adapters (future story).

3. **NAV contract stubs**: The `datasource.internal.adapters.nav` package defines the NAV Online Szamla API contract (interfaces + records) based on the real NAV v3 XSD schemas, ready for implementation when credentials are available.

**What is NOT in this story (deferred):**
- Actual NAV Online Szamla XML/JAXB integration (requires technical user credentials)
- SHA-512 password hashing and SHA3-512 request signature
- WireMock NAV simulator for integration testing
- JAXB code generation from NAV XSD schemas
- Credential management UI and encrypted storage


**The call flow after this story:**

    ScreeningController.searchPartner(taxNumber)
      -> ScreeningService.search(taxNumber)
          -> TX1: idempotency guard + create empty snapshot
          -> DataSourceService.fetchCompanyData(taxNumber)   <-- RENAMED from ScrapingService
              -> CompanyDataAggregator.aggregate(taxNumber)
                  -> [StructuredTaskScope virtual thread]
                      -> DemoCompanyDataAdapter.fetch(taxNumber)   <-- NEW (replaces 3 fictional adapters)
          -> TX2:
              -> ScreeningRepository.updateSnapshotData(...)
              -> SnapshotDataParser.parse(snapshotData)            <-- UPDATED for "demo" key
              -> VerdictEngine.evaluate(parsed, checkedAt, cfg)
              -> ScreeningRepository.createVerdict(id, status, confidence)
              -> writeAuditLog
          -> publish PartnerStatusChanged (if changed)
          -> publish PartnerSearchCompleted
          -> return SearchResult with real verdict

### DANGER ZONES -- Common LLM Mistakes to Avoid

1. **DO NOT leave any reference to the old scraping package.** After the rename, grep the entire codebase for "scraping" — there should be ZERO hits in Java source or YAML config (story markdown files and git history are OK). This includes: package statements, import statements, Javadoc, log messages, property keys, and ArchUnit test rules.

2. **DO NOT change CompanyDataAggregator's StructuredTaskScope pattern.** The parallel orchestration, Joiner.awaitAll(), tenant propagation via withTenant(), and timeout handling are all correct from Story 2.2. Move the file, update package/imports, update the properties call from getScraping() to getDataSource() — nothing else.

3. **DO NOT make DemoCompanyDataAdapter a @Component.** It must be registered as a bean ONLY through the DataSourceModeConfig @Configuration class using @ConditionalOnProperty. The old adapters were @Component (always registered) — the new pattern is profile-driven.

4. **DO NOT produce ScrapedData with available=false from the demo adapter.** Demo mode simulates a world where all data sources are healthy. The adapter always returns available=true. The JSONB structure must include the "available": true field so SnapshotDataParser produces sourceAvailability = AVAILABLE.

5. **DO NOT forget to update CRITICAL_SOURCES in SnapshotDataParser.** The current list is ["nav-debt", "e-cegjegyzek", "cegkozlony"]. After this story, when mode is demo, the parser must recognize the "demo" adapter key. The cleanest approach: make the parser dynamic — it reads whatever keys exist in the JSONB map rather than hardcoding source names.

6. **DO NOT change VerdictEngine or its tests.** The engine is a pure function that operates on SnapshotData records. It does not care where the data came from. Leave it untouched.

### SnapshotDataParser Update Strategy

The current SnapshotDataParser hardcodes three adapter keys ("nav-debt", "e-cegjegyzek", "cegkozlony") and extracts specific fields from each. The demo adapter provides ALL risk signals in a single "demo" entry.

**Recommended approach -- make the parser dynamic:**

Instead of hardcoding adapter names, iterate over all keys in the JSONB map. For each adapter entry:
1. Check the "available" field to set sourceAvailability
2. Check "taxNumberStatus" == "SUSPENDED" to set taxSuspended
3. Check "hasPublicDebt" == true to set hasPublicDebt
4. Check "hasInsolvencyProceedings" == true to set hasInsolvencyProceedings

This means the parser works with ANY adapter key name -- "demo", "nav-debt", future "nav-m2m", etc. -- without code changes. The only contract is the field names inside the adapter data.

**Backward compatibility:** Old JSONB data stored with keys "nav-debt", "e-cegjegyzek", "cegkozlony" will still parse correctly since the dynamic approach reads whatever keys exist.

**Expected demo JSONB structure:**

    {
      "demo": {
        "available": true,
        "companyName": "Pelda Kereskedelmi Kft.",
        "registrationNumber": "01-09-123456",
        "taxNumberStatus": "ACTIVE",
        "hasPublicDebt": false,
        "hasInsolvencyProceedings": false,
        "debtAmount": 0,
        "debtCurrency": "HUF",
        "status": "ACTIVE"
      }
    }


### NAV Online Szamla API Technical Context (for stub design)

The stub interfaces and records in Task 5 should accurately model the real NAV v3 API contract. Key technical details from the API specification:

**API endpoints (v3.0):**
- Production: https://api.onlineszamla.nav.gov.hu/invoiceService/v3/{operation}
- Test: https://api-test.onlineszamla.nav.gov.hu/invoiceService/v3/{operation}
- Protocol: XML over HTTPS (NOT JSON)
- XSD schemas: invoiceApi.xsd, invoiceData.xsd, invoiceBase.xsd, invoiceAnnulment.xsd, serviceMetrics.xsd

**Operations modeled by stubs:**
- QueryTaxpayer: verify partner tax number, get name/address/incorporation type
- QueryInvoiceDigest: list invoices by date range + direction (paginated)
- QueryInvoiceData: get full invoice with line items (VTSZ codes for EPR)

**Authentication model (for future implementation):**
- login (technical user name) -- encrypted in DB
- password -- SHA-512 hashed, sent as passwordHash
- signing key (cserekulcs) -- encrypted in DB, used for SHA3-512 requestSignature
- tax number -- identifies the company

**Key design decision:** QueryTaxpayer is NOT representation-bound -- any technical user can query any tax number. This makes it usable for partner screening without needing the accountant's EGYKE representation.
